1. Análisis de Catálogo y Ventas

Identificación de Top Ventas y Rentabilidad

Metodología y Supuestos de Análisis

1. Filtro Financiero

Solo estado "delivered". Bajo una perspectiva financiera estricta, únicamente los pedidos entregados (96.478) representan ingresos reales. Se descartaron pedidos cancelados o en tránsito.

2. Manejo de Nulos (Fallback)

Se consolidó el catálogo de 110.197 ítems vendidos aplicando una jerarquía: Traducción EN > Nombre PT > 'unknown_category'. Esto garantizó preservar el 100% de la data en el análisis sin pérdida por metadatos faltantes.

3. Ingresos sin Flete

La métrica de ingresos netos se calculó exclusivamente usando el precio unitario del producto. Se excluyó el valor del flete al ser un costo logístico y no el precio real del ítem, asegurando precisión en la rentabilidad.

Conclusión Ejecutiva: El Verdadero Foco Crítico

  • Logística Transversal

3. Segmentación de Clientes

Identificación de Clientes Fieles vs Riesgo (Modelo RFM + Clustering)

Segmentación de Clientes: El Reto Estructural

Framework Base: RFM

  • El problema de K-Means Puro

    Al analizar los 93.358 clientes únicos, se descubrió que el 97.0% tiene una Frecuencia = 1 (compradores de una sola vez).

  • Varianza Nula

    K-Means fallaba al segmentar porque la variable Frequency no tenía suficiente varianza. El algoritmo solo dividía por Gasto Monetario, creando grupos inútiles para medir "fidelidad".

Nivel 1: Regla de Negocio (Frecuencia)

Separación inicial rígida:
• F >= 2: Compradores recurrentes (3.0%)
• F = 1: Compradores únicos (97.0%)

Nivel 2: Clustering K-Means (R + M)

Aplicación de K-Means (k=2) dentro de cada grupo sobre las variables de Recency y Monetary, optimizado mediante Silhouette Score.

4. Arquitectura del Motor de Recomendación

Despliegue Predictivo en Tiempo Real (<100ms) y Escalabilidad

Principio de Diseño y Flujo de Datos

El modelo predictivo no calcula todo en tiempo real.
Las recomendaciones maestras se pre-computan offline y se sirven como lecturas de caché ultra-rápidas. El pipeline online solo orquesta la extracción y aplica personalización de último milisegundo.

1. Request Web

El cliente navega. Identificación de sesión activa (Customer ID o ID Anónimo).

2. Lookup Caché (Redis)

Extracción en <5ms del Perfil de Segmento (RFM), Historial de Compras y Vistas.

3. Two-Stage Engine

Generación de Candidatos Iniciales + Algoritmo de Ranking Dinámico.

4. Renderizado Web

Devolución de los Top-N productos mediante API REST (FastAPI) al frontend.

Two-Stage Recommender (Arquitectura en 2 Etapas)

Etapa 1: Candidate Generation
(Reducción del espacio de búsqueda)

  • Filtro por Segmento (RFM)

    Se toma el cluster_id del cliente y se recupera un pool genérico de los productos más relevantes para ese nicho.

  • Pool Pre-computado

    Se obtienen 50-200 productos candidatos pre-calculados, excluyendo dinámicamente aquellos ítems que ya compró.

  • Propósito del Filtro

    Evitar iterar sobre los 32.000 productos del catálogo. Mejora radicalmente la latencia para la siguiente etapa.

Etapa 2: Ranking Dinámico
(Personalización de último milisegundo)

Reto de Esparsidad (Olist)

Dado que el 97% de los clientes compró 1 sola vez, algoritmos de matriz fallarían. Se requiere enrutamiento adaptativo.

Escenario del ClienteAlgoritmo Aplicado
Historial Rico (>1 compra)Item-based CF (Co-ocurrencia)
Historial Pobre (1 compra)Content-Based (Features producto)
Cold Start (Nuevo / Anónimo)Top Global / Categoría Popular

Señales de Entrada y Stack Tecnológico

Señales Implícitas (Vitales)

En un dataset con alta esparsidad, el comportamiento de navegación multiplica exponencialmente el volumen de datos por usuario.

  • Vistas de productos no comprados.
  • Tiempo de permanencia.
  • Carritos abandonados.

Pipeline Offline y Feedback Loop

Pipeline Batch (Ej. Cada 24h)

1. Update: Reasignación dinámica de clientes a clusters.
2. Re-compute: Cálculo de productos top por segmento.
3. Matrices: Actualización de similitudes (Item-based CF).
4. Publish: Escritura atómica en caché para evitar downtime.

El Feedback Loop Continuo

Un motor estático no aprende. El sistema captura la interacción de cada recomendación servida para inyectar este feedback explícito en el siguiente ciclo de reentrenamiento, evaluando directamente métricas como CTR y Conversion Rate.

Paso 2: Cálculo del Punto Óptimo (São Paulo)

Método: Centroide Ponderado

El objetivo no es el centro geográfico simple, sino minimizar la distancia a los clientes dando mayor peso (gravedad) a los clientes con mayor gasto histórico.

Fórmula Aplicada

lat_óptima = Σ(lat_i × ingresos_i) / Σ(ingresos_i)
lng_óptima = Σ(lng_i × ingresos_i) / Σ(ingresos_i)

Decisiones de Diseño de Prompts (Prompt Engineering)

Solo 1 Skill Integrada

El agente tiene una única skill (check_order_status). Las skills deben reservarse para API calls (datos en vivo). La recomendación y la empatía son capacidades nativas de razonamiento del modelo; es un error tratar de resolverlas con lógicas deterministas (if/else).

Briefing > Reglas Numeradas

Los modelos de producción (Claude, GPT) razonan mejor desde un contexto rico (Persona) y un objetivo claro que desde un checklist rígido. Un briefing permite que el agente conserve su capacidad de adaptación si el cliente dice algo inesperado.

Contexto en Formato JSON

Inyectar el perfil del cliente en JSON elimina ambigüedades. Permite al LLM distinguir claramente variables y razonar sobre estructuras anidadas (ej. la experiencia negativa previa o el catálogo de productos) sin confundirlas con instrucciones del sistema.
